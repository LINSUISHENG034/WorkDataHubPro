<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| failed checkpoint selection -> compatibility-case payload | If the checkpoint-to-payload mapping is wrong, adjudication records become semantically false even when `primary_failure` is correct. |
| shared runtime helper -> explicit slice runners | A shared helper must not flatten domain meaning; runners still own the actual payload maps they pass in. |
| replay tests -> governance evidence truthfulness | Without replay assertions on persisted cases, the repo can claim explainability while storing misleading evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03.1-01 | T | compatibility-case payload mapping | mitigate | Replace the `reference_derivation` / `monthly_snapshot` fallback branch with an exact helper keyed by the failed checkpoint name and cover all required checkpoints in replay tests. |
| T-03.1-02 | R | persisted adjudication evidence | mitigate | Assert in replay tests that `sample_locator`, `legacy_result`, and `pro_result` all match the same failed checkpoint named by `primary_failure.checkpoint_name`. |
</threat_model>
