## Summary

The audit findings are real code-level defects, not governance-only wording issues. The three replay slice runners still special-case only `reference_derivation` and otherwise build `CompatibilityCase` payloads from `monthly_snapshot`, even when the first failed checkpoint is `fact_processing`, `identity_resolution`, or `contract_state`. `src/work_data_hub_pro/apps/orchestration/replay/diagnostics.py` and `src/work_data_hub_pro/governance/evidence_index/file_store.py` also resolve manifest-declared package paths without enforcing that they stay inside the current comparison-run package, and `src/work_data_hub_pro/apps/etl_cli/main.py` catches missing-run diagnostics but not invalid path-like identifiers.

The right implementation strategy is narrow remediation, not replay-surface expansion. Phase 3 already established the registry-backed CLI and file-backed package model. Phase 03.1 should harden those exact contracts without duplicating planned Phase 4 lookup, redaction, or lifecycle work.

**Primary recommendation:** plan this slice as three plans across two waves:
1. replay-slice truthfulness fixes plus per-domain replay coverage for failed intermediate checkpoints;
2. diagnose package-path boundary enforcement plus typed invalid-id CLI coverage;
3. Phase 3/03.1 verification and status-artifact regeneration after executed evidence is green.

## Project Constraints

- Preserve capability-first boundaries from the architecture blueprint: keep slice-specific replay behavior in explicit runners, evidence read/write rules in `governance`, and CLI presentation in `apps`.
- Keep the comparison-run package as the single governed diagnostics source of truth; do not introduce a second diagnostics schema or a global run index.
- Use `uv` for verification commands and require `uv run pytest -v` before any completion claim.

## Architecture Patterns

### Pattern 1: Checkpoint-specific compatibility-case payload selection

**What:** factor each slice runner's failure-case payload selection into an explicit mapping from `primary_failure.checkpoint_name` to the matching baseline file and observed payload.

**Why:** today the run report names the correct failed checkpoint, but the persisted compatibility case points reviewers to `monthly_snapshot` evidence for most failures.

**Recommended shape:**
- `reference_derivation` keeps its existing explicit payload path
- add explicit payload branches for `fact_processing`, `identity_resolution`, and `contract_state`
- keep `monthly_snapshot` as the projection-specific branch, not the fallback for every non-derivation failure
- if `source_intake` remains warning-only/contract-only in this slice, do not invent a fake compatibility case for it

### Pattern 2: Fail-closed comparison-run package resolution

**What:** enforce one helper that resolves manifest `package_paths` relative to the current evidence root and rejects:
- absolute paths
- `..` escapes
- resolved paths outside the current comparison-run package or evidence root

**Why:** the current file-backed diagnose path is intended to explain one completed comparison run, not follow arbitrary manifest-controlled file pointers.

**Recommended placement:** keep the path-boundary enforcement adjacent to `FileEvidenceIndex` path resolution so both direct readers and `load_replay_diagnostics(...)` use the same guard.

### Pattern 3: Typed CLI boundary for invalid identifiers

**What:** treat invalid `comparison_run_id` input the same way the CLI already treats missing comparison runs: emit a short typed message and exit non-zero without a traceback.

**Why:** `replay diagnose` is part of the agent-facing Phase 3 contract. Invalid user input must not downgrade into implementation-level noise.

**Recommended placement:** catch `ValueError` at the `replay_diagnose` CLI boundary and preserve the existing machine-readable success path for valid runs.

## Recommended Plan Split

### Plan 01: Truthful failed-checkpoint compatibility cases
- update all three replay slice runners to select compatibility-case payloads by failed checkpoint
- extend replay tests so `fact_processing`, `identity_resolution`, and `contract_state` failures assert both `primary_failure.checkpoint_name` and the persisted compatibility-case payload/path match the same checkpoint

### Plan 02: Diagnose hardening and CLI stability
- harden manifest package-path resolution in `FileEvidenceIndex` and `load_replay_diagnostics(...)`
- add contract coverage for absolute-path and escaping-relative-path manifests
- add CLI-level contract coverage for invalid path-like `comparison_run_id` values and catch the `ValueError` path in `main.py`

### Plan 03: Verification and governance status sync
- refresh Phase 3 verification artifacts and any linked planning/wiki status docs so they reflect "governance sign-off restored after remediation" only after the new evidence passes
- keep this plan doc/test-driven where possible; use exact `rg`/pytest commands for artifact truthfulness rather than broad narrative updates
