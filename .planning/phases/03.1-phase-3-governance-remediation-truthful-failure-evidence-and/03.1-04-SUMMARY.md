### Deferred Issues

**1. [Rule 2 - Architecture] Compatibility case uses CURRENT baseline file instead of captured baselines**

- **Found during:** Task 1 verification
- **Issue:** After `_bootstrap_intermediate_baselines` correctly returns `outcome.intermediate_payloads`, the `test_*_replay_uses_failed_checkpoint_payloads` tests still fail with `AssertionError` (not `TypeError`). The assertion `compatibility_case.legacy_result["rows"] == baselines["fact_processing"]` fails because `compatibility_case.legacy_result["rows"]` contains FORCED-FAIL data (the corrupted baseline currently on disk) instead of the original bootstrapped baselines.
- **Root cause:** When `run_annual_award_slice` builds the compatibility case, it loads `expected_fact_processing` from the CURRENT baseline file on disk (`_load_replay_baseline`). When the test writes FORCED-FAIL data to the baseline file and calls `run_annual_award_slice` again, the slice loads FORCED-FAIL as `expected_fact_processing`, which ends up in `legacy_payloads["fact_processing"]` passed to `build_failure_compatibility_case_payload`. The reference implementation in `test_phase2_event_domain_gates.py` passes because its `_bootstrap_intermediate_baselines(runner, workbook_path, replay_root)` has a different structure that works correctly.
- **Files affected:** `src/work_data_hub_pro/apps/orchestration/replay/annual_award_slice.py`, `annual_loss_slice.py`, `annuity_performance_slice.py`
- **Fix needed:** The slice implementations need to use the bootstrapped baselines (captured at bootstrap time) for building the compatibility case, not the current baseline files on disk which may have been modified by test code. This likely requires passing the captured baselines as a parameter or storing them in the execution context.
- **Commit:** `11a7f90`

## Reference Implementation Comparison

The reference implementation in `tests/replay/test_phase2_event_domain_gates.py` has `_bootstrap_intermediate_baselines(runner, workbook_path, replay_root) -> dict[str, list[dict[str, object]]]` with `return outcome.intermediate_payloads` and passes all tests. The difference is architectural: the reference takes `runner` as a parameter (supporting both award and loss runners from a single helper), while the slice files hardcode the specific slice runner. However, this architectural difference does not fully explain why the slice tests fail while the reference passes - both use the same pattern of capturing baselines and calling the runner a second time with FORCED-FAIL data.
