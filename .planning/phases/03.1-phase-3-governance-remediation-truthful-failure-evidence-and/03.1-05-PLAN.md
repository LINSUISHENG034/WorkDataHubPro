<objective>
Fix truthful failure evidence compatibility-case payloads to use in-memory intermediate_payloads from the outcome, not file-system reads that can be corrupted by the test.

Purpose: When a test corrupts a baseline file after bootstrap, the compatibility_case must still reference the correct baseline (from outcome.intermediate_payloads), not the corrupted file content.
Output: All three slice runners pass outcome.intermediate_payloads as legacy_payloads to build_failure_compatibility_case_payload.
</objective>

<context>
The issue is in how each slice runner constructs `legacy_payloads` for `build_failure_compatibility_case_payload`. Currently each runner loads `expected_*` from the file system via `_load_replay_baseline`. If a test corrupts the file between bootstrap and the failed run, the corrupted data ends up in `legacy_payloads`.

The fix: pass `outcome.intermediate_payloads` directly as `legacy_payloads`. This dict was captured during the bootstrap run and contains the correct baseline data regardless of subsequent file mutations.

Current (broken) pattern in all three slices:
```python
expected_fact_processing = _load_replay_baseline(fact_processing_baseline_path, "fact_processing")
...
legacy_payloads = {
    "fact_processing": expected_fact_processing,  # <- reads from file, can be corrupted
    ...
}
```

Correct pattern:
```python
legacy_payloads = outcome.intermediate_payloads  # <- in-memory, not affected by file corruption
```
</context>
