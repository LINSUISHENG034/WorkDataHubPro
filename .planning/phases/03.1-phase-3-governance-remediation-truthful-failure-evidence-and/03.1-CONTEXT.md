# Phase 03.1: Phase 3 governance remediation - truthful failure evidence and diagnose hardening - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning
**Source:** `docs/gsd/reviews/2026-04-13-phase3-completion-audit-review.md`

<domain>
## Phase Boundary

Phase 03.1 is a narrow remediation slice inserted after Phase 3 to restore trust in the existing replay failure and diagnostics surface before governance sign-off.

This phase must correct only the concrete completion-audit blockers:

- compatibility-case evidence must point to the actual failed checkpoint payload rather than falling back to `monthly_snapshot`
- `replay diagnose` must reject manifest-controlled package paths that escape the current comparison-run evidence root
- invalid `comparison_run_id` CLI input must return a short typed error instead of a raw traceback

This phase is not a general replay redesign and it must not absorb broader Phase 4 lookup, redaction, or compatibility lifecycle scope.
</domain>

<decisions>
## Implementation Decisions

### Locked Requirement Anchors
- Treat this remediation as re-closing `PAR-02`, `PAR-03`, `PIPE-03`, and `OPS-01`.
- Do not treat this phase as a new requirement expansion into `OPS-03`, `GOV-01`, or `GOV-03`; those remain owned by Phase 4 plans.

### Locked Scope
- Preserve the current shared replay runtime, registry-backed replay CLI, typed setup failures, and opaque temp-id policy; the audit does not reopen those accepted areas.
- Keep the fix surface constrained to truthful failure evidence and diagnose-path hardening in the existing Phase 3 runtime/CLI contracts.
- Reuse the current comparison-run package model and registry-backed run resolution; do not introduce a database index, new storage backend, or a changed `comparison_run_id` format.

### Finding 1: Truthful compatibility-case evidence
- Update all three replay slice runners so any blocking checkpoint can produce a checkpoint-specific `CompatibilityCase` payload.
- The `compatibility_case.sample_locator`, `legacy_result`, and `pro_result` fields must reference the same failed checkpoint named by `primary_failure.checkpoint_name`.
- Add replay coverage that deliberately fails `fact_processing`, `identity_resolution`, and `contract_state` and asserts the stored compatibility case points at the matching failed-checkpoint evidence.

### Finding 2: Diagnose package-path boundary enforcement
- `load_replay_diagnostics(...)` and `FileEvidenceIndex` must resolve manifest-declared package paths only within the current comparison-run package root or evidence root.
- Reject absolute paths and escaping relative paths in manifest package-path resolution unless a future spec explicitly allows them.
- Add contract coverage for path-escape attempts so diagnose cannot be redirected to files outside the governed comparison-run package.

### Finding 3: Typed invalid-id CLI behavior
- Catch invalid `comparison_run_id` values such as path-containing input at the CLI boundary for `replay diagnose`.
- Emit the same short typed error style already used for missing comparison runs instead of surfacing a Python traceback.
- Add CLI-level contract coverage for invalid identifiers; loader-level validation alone is not sufficient.

### Planning Guardrails
- Do not duplicate planned Phase 4 work for `replay lookup`, evidence redaction, compatibility-case lifecycle commands, or canonical runbook additions.
- If a fix touches code that Phase 4 also plans to extend, constrain the change to the minimum truthful Phase 3 contract needed now and leave broader expansion to Phase 4.
- Regenerate verification and governance artifacts from executed evidence after the behavior changes land; do not leave Phase 3 status docs claiming unconditional closure without reflecting the remediation.
</decisions>
