<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| accepted replay assets -> replay gate execution | Missing or stale baseline assets would let intermediate checkpoints regress into self-compare or non-truthful behavior. |
| source-intake contract expectations -> gate status | If `source_intake` expectations are derived from current runtime rows, the contract gate becomes another self-compare placeholder. |
| intermediate checkpoint failures -> compatibility evidence | If the first divergent checkpoint is not surfaced, investigators will be forced to diagnose drift only at the final snapshot. |
| committed verification-asset registry -> accepted checkpoint baseline governance | If new checkpoint baselines are not registered, the repo will regain a hidden protection mechanism outside the declared asset model. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-04 | T | promoted intermediate checkpoints | mitigate | Load repo-native accepted baseline files for `fact_processing`, `identity_resolution`, and `contract_state` in all accepted slices. |
| T-02.1-05 | R | `source_intake` contract gate | mitigate | Use explicit fixed expectations such as `record_count`, `required_fields`, and `allowed_adaptations` rather than current payload self-comparison. |
| T-02.1-06 | D | replay truthfulness evidence | mitigate | Add tests that force intermediate failures before `monthly_snapshot` and assert missing-baseline failure for `reference_derivation`. |
| T-02.1-10 | R | checkpoint baseline governance | mitigate | Register the new accepted checkpoint baseline assets in `reference/verification_assets/phase2-accepted-slices.json` and guard them with contract tests. |
</threat_model>
